# CoreCommerce — План розвитку та поточний стан

> Рушій інтернет-магазину на чистому PHP та MySQL.
> Документ оновлено відповідно до реального стану кодової бази.

---

## Поточний стан реалізації вимог

### 1. Підтримка зміни мови інтерфейсу ✅ (Реалізовано)

**Що зроблено:**
- `LocalizationManager` зберігає вибір мови в `$_SESSION['lang']` і cookie (365 днів), читає `Accept-Language` браузера
- Мовні файли `lang/ua.php` і `lang/en.php` (~380+ ключів кожен), helper `__()`
- Перемикач мови в шапці сайту (`/language/{lang}`)
- Весь UI адмінки та фронтенду через `__()` — жодного хардкодного тексту
- JS-рядки (alert, toast, confirm) через `window.LANG` об'єкт у layout

**Архітектурне рішення по SEO і контенту:**
SEO-поля (`meta_title`, `meta_description`) зберігаються в БД однією мовою (українською).
Локалізація контенту (різні назви товарів для UA/EN) — **поза поточними вимогами**.
Переклад назв на рівні БД реалізується окремо через таблицю `entity_translations` якщо знадобиться.

**Що залишилось:**
- Адмінка лише українською (перемикач мови в адмінці не реалізовано)

---

### 2. Підтримка зміни дизайну ✅ (Реалізовано)

**Що зроблено:**
- `ThemeManager`, теми `default` і `modern` у `resources/themes/`
- Активна тема зберігається в налаштуваннях, перемикається в адмінці (`/admin/themes`)
- Кожна тема має власний `layout.php` з повним набором SEO-мета-тегів (OG, canonical, robots)

**Що залишилось:**
- Тема `modern` неповна (не всі сторінки)
- Немає `theme.json` контракту з перевіркою сумісності версій

---

### 3. Адаптивний дизайн ⚠️ (Частково)

**Що зроблено:**
- Базові media-query у темі `default`
- Bootstrap-подібна сітка

**Що залишилось:**
- Мобільне меню потребує доопрацювання
- Адмінка не повністю адаптивна на < 768px

---

### 4. ЧПУ адреси ✅ (Реалізовано)

**Що зроблено:**
- `Router` з підтримкою slug-параметрів
- `SlugHelper` з генерацією, перевіркою унікальності, 301-редиректами
- ЧПУ для товарів (`/products/{slug}`), категорій (`/category/{slug}`), сторінок (`/pages/{slug}`)
- 404 повертає коректний HTTP 404 статус

**Що залишилось:**
- Вкладені категорії (`/category/parent/child`) — міграція `path`-колонки підготовлена
- XML sitemap не генерується

---

### 5. Система плагінів ✅ (Реалізовано)

**Що зроблено:**
- `PluginManager`, `PluginInterface` з `getSettingsSchema()`
- `PluginDB` — sandbox-проксі (дозволяє запис лише в таблиці з префіксом `plugin_{slug}_`)
- `PluginSecurityException`
- Семантична валідація версій залежностей (`requires` в `info.json`)
- Хуки `do_action()` / `apply_filters()`
- Налаштування плагінів в адмінці (`/admin/plugins/settings/{slug}`)
- Вбудовані плагіни: `LiqPayGateway`, `TestPlugin`
- Подія `do_action('order.created')` при створенні замовлення

**Що залишилось:**
- Документація для розробників плагінів

---

### 6. Автоматична система оновлень ⚠️ (Частково)

**Що зроблено:**
- `config/updater.php` з каналами оновлень
- Вкладка "Оновлення" в налаштуваннях з чек-листом готовності
- Backup перед оновленням

**Що залишилось:**
- Update-сервер — заглушка (`https://api.corecommerce.com/v1/update`)
- Немає авто-перевірки нових версій за розкладом
- Немає rollback при невдалому оновленні

---

### 7. Система безпеки ✅ (Реалізовано)

**Що зроблено:**
- CSRF-токени на всіх формах
- Rate limiting логіну (IP + email) через `LoginRateLimiter` і таблицю `login_attempts`
- MIME-валідація завантажень через `finfo`
- `HttpOnly` / `Secure` cookie-флаги
- `DB::$pdo` — приватний з публічними static-обгортками
- Підписані вебхуки (HMAC-SHA256) для Prom.ua
- `PluginSecurityException` — sandbox на рівні БД
- Логування дій адміністраторів

**Що залишилось:**
- CSP/HSTS заголовки не додані
- Rate limiting лише на логін (не на інші форми)

---

### Категорії та фільтрація ✅ (Реалізовано)

**Що зроблено:**
- Багаторівневе дерево категорій (`parent_id`, `getTree()`, `getBreadcrumbs()`)
- `Attribute`, `ProductAttribute`, `ProductAttributeValue` — прив'язка атрибутів до категорій
- `ProductFilterService` — фільтрація за ціною, атрибутами, категорією з дочірніми; AJAX-оновлення
- `Paginator` — пагінація з ±2 вікном і крапками, зберігає GET-параметри

---

### SEO ✅ (Реалізовано)

**Що зроблено:**
- `SeoService` — централізована логіка SEO для всіх типів сторінок
- Пріоритет: `seo_settings` (БД) → `meta_*` поля сутності → шаблон → fallback
- Шаблони з масками `{name}`, `{price}`, `{category}`, `{shop_name}`, `{slug}`
- Шаблони для товарів, категорій, сторінок і головної — в налаштуваннях адмінки
- OG-теги, canonical, robots, keywords в обох темах
- Таблиця `seo_settings` для перевизначення SEO будь-якої сутності

## Пріоритети наступних кроків

### Висока пріоритетність
1. **CSP/HSTS заголовки** — фінальний крок безпеки (Security вимога #7)
2. **Вкладені ЧПУ категорій** — міграція `path`-колонки вже підготовлена
3. **Тема `modern`** — завершення адаптації всіх сторінок

### Середня пріоритетність
4. **Мобільне меню** — адаптивність адмінки
5. **XML sitemap** — автогенерація для SEO
6. **Rate limiting** — розширити на форми реєстрації, відгуків, підписки

### Низька пріоритетність (майбутній розвиток)
7. **Update-сервер** — реальна перевірка нових версій
8. **Rollback оновлень** — атомарний відкат
9. **`entity_translations`** — якщо знадобиться мультимовний контент

## Архітектурні принципи проекту

- **Без важких фреймворків** — чистий PHP, лише PHPMailer як зовнішня залежність
- **Без curl** — HTTP-запити через `file_get_contents` + stream context
- **Безпека шарами** — rate limiting, MIME-валідація, sandbox БД для плагінів, підписані вебхуки
- **Зворотна сумісність** — нові системи не ламають існуючий код (напр. LiqPay перевикористовує збережені ключі)
- **Явні сервіс-класи** — `SeoService`, `PromApiClient`, `BankCurrencyService`, `PaymentManager`, `MailService`
- **Єдиний патерн адмінки** — пагінація + фільтрація + bulk-actions скрізь
